## 🔄 完整执行顺序

```
Middleware → Guards → Interceptors (Before) → Controller/Service → Interceptors (After) → Response
                  ↓                                    ↓
              Filters ←————————————— (异常捕获) ←————————┘
```

具体数字排序如下：

1. **Guards（守卫）** - _最先执行_
    - 负责权限验证、角色检查等
    - 如果守卫抛出异常或返回 `false`，请求**立即终止**，后续的拦截器和控制器都不会执行
2. **Interceptors - Before Logic（拦截器前置逻辑）**
    - 在守卫通过后、控制器方法执行前运行
    - 可用于参数转换、日志记录、缓存判断等
3. **Controller / Service（业务逻辑）**
4. **Interceptors - After Logic（拦截器后置逻辑）**
    - 控制器返回结果后，响应发送给客户端前运行
    - 可用于响应数据格式化、添加额外字段等
5. **Filters（异常过滤器）** - _特殊位置_
    - **不参与正常流程的顺序链**
    - 只在上述**任何阶段**抛出异常时被触发
    - 可以捕获来自 Guard、Interceptor、Controller 等所有层的异常

### 🔍 核心纠正

1. **Interceptor 并不"处理"异常**（除非你主动用 `catchError`）
    - Interceptor 的 `intercept()` 方法返回的是一个 `Observable`
    - 如果下游（Controller/Service）抛出异常，这个 Observable 会进入 **error 通道**
    - 如果你**没有**在 Interceptor 里写 `catchError()`，异常会**直接穿透** Interceptor，交给 Filter 处理
    - 只有当你显式写了 `catchError(err => ...)` 时，Interceptor 才算"处理"了异常（此时 Filter **不再触发**）
2. **Filter 不是在 Interceptor "之后"触发，而是在 Interceptor "外层"**
    - 正确的模型是**洋葱模型 / 嵌套捕获**：

Filter (最外层异常捕获)
