| 维度          | Java/Spring                             | Node/NestJS              |
| ----------- | --------------------------------------- | ------------------------ |
| 历史起点        | .properties（扁平 key=value）→ 升级到 yml 解决层级 | dotenv（扁平 key=value）→ 够用 |
| 部署文化        | 打 jar + 外部配置文件，运维改 yml 不重打包             | 容器化，env 注入               |
| process.env | 无强文化，不是主要配置源                            | 核心机制，事实标准                |
| 配置复杂度       | Spring 的 profile/多数据源/bean 图谱天然层级深      | 相对扁平                     |
| 约定          | application.yml 是 Spring Boot 默认        | .env 是社区默认               |

本质：不是"yml 更好"，而是两个生态的部署模型和配置文化不同。 Java 应用打包后配置和代码分离，yml 作为外部文件适合运维直接改；Node 应用跑在容器里，env 是云原生的通用语言。

### 配置的本质

配置 = "随环境变化但不随代码变化的值"。分两类，应该用不同载体：

| 配置类型  |	 例子    |   	推荐载体	    |     理由  |
| ---- | ---- | ---- | ---- |
| 敏感 + 部署相关 |	DB_PASSWORD、REDIS_URL、JWT_SECRET  | .env	| 不进版本库；容器原生注入；12-Factor |
| 结构化 + 非敏感 |	日志路径、功能开关、嵌套业务参数      |  yml	| 层级清晰；可进版本库；多环境用 profile 切 |
