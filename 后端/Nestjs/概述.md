NestJS 是一个深受 **Angular** 架构启发并借鉴了 **Java Spring** 设计思想的 Node.js 框架。在全栈学习中，理解其核心原则比单纯记忆 API 更重要。
以下是 NestJS 的核心设计原则与最佳实践总结：

### 1. 核心架构原则

- **模块化 (Modularity)**
    - **原则**：按**业务领域**（Feature）而非技术层次划分模块。每个模块应是一个内聚的业务单元（包含 Controller, Service, Entity, DTO）。
    - **实践**：使用 `@Module()` 装饰器封装功能。避免创建巨大的 "God Module"，提倡共享模块（Shared Modules）和动态模块（Dynamic Modules）来处理跨模块依赖。
- **依赖注入 (Dependency Injection / IoC)**
    - **原则**：**控制反转**。类不应自己创建依赖对象，而应由容器注入。这是实现解耦和可测试性的基石。
    - **实践**：优先使用**构造函数注入**；通过 `@Injectable()` 标记服务；利用 Token 或接口抽象来注入具体实现，便于单元测试时替换 Mock 对象。
- **分层架构与关注点分离 (Separation of Concerns)**
    - **Controller 层**：仅负责 HTTP 请求路由、参数接收、调用 Service 和返回响应。**绝不包含业务逻辑**。
    - **Service 层**：承载核心业务逻辑、事务控制和规则判断。
    - **Repository/Provider 层**：抽象数据访问细节，屏蔽底层 ORM（TypeORM/Prisma）差异。
    - **DTO (Data Transfer Object)**：严格隔离外部输入与内部领域模型，防止过度暴露字段，配合 `class-validator` 进行校验。
- **面向切面编程 (AOP)**
    - NestJS 提供了丰富的 AOP 机制来处理横切关注点，避免业务代码污染：
        - **Middleware**：处理原始请求/响应（如日志、CORS）。
        - **Guards**：权限认证与访问控制。
        - **Interceptors**：响应转换、缓存、性能监控。
        - **Pipes**：参数验证与类型转换。
        - **Exception Filters**：统一异常处理与错误格式化。

### 2. 开发最佳实践

| 维度         | 推荐做法                                            | 避免做法                     |
| ---------- | ----------------------------------------------- | ------------------------ |
| **API 设计** | 遵循 RESTful 规范，正确使用 HTTP 动词与状态码                  | 所有接口都用 POST，状态码全返 200    |
| **配置管理**   | 使用 `@nestjs/config` + `yml`，通过 ConfigService 注入 | 硬编码配置，直接使用 `process.env` |
| **数据库**    | 使用 Repository 模式或 QueryBuilder，定义清晰的 Entity 关系  | 在 Service 中直接写原生 SQL     |
| **异步处理**   | 全面拥抱 RxJS (Observables) 或 async/await           | 混用回调地狱，阻塞主线程             |
| **测试**     | 编写单元测试（Mock 依赖）+ E2E 测试                         | 只手动测试，不写自动化用例            |
| **文档**     | 集成 Swagger/OpenAPI (`@nestjs/swagger`)          | 手写文档且不与代码同步              |
